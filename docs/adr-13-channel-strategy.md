# ADR-13 - Channel strategy v2: official APIs first, mirrored channels, daemon-assisted delivery

**Status:** Accepted · 2026-07-03 to 2026-07-09 · extends [ADR-06](decisions.md#adr-06---delivery-surface) and [ADR-12](decisions.md#adr-12---taxonomy-rename-the-delivery--notification-contract-and-fleet-health-hardening) · extended by [ADR-15](decisions.md#adr-15---gated-engine-migration-marker-table-integrity-and-weekly-convergence). The companion patterns adopted the same week are recorded at the bottom; a dated addendum (2026-07-08 to 2026-07-09) follows them.

**Context.** ADR-12 made delivery uniform: chat + file surfaces + one tagged text message per producer. Three pressures then hit the channel layer at once. (1) The second messaging channel ran on a community bridge, an unofficial client for a consumer messaging platform. Platform terms bar unofficial clients and automated messaging, and the risk model is worse than it looks: enforcement scores accumulated account history (linked-device fingerprint, automated send volume, IP churn), so a penalty can land while the bridge is off, and it lands on the human's account, not on the automation. (2) Rich artifacts (HTML reports, boards, documents) had no first-class carrier; the OS-native text thread is plain text only. (3) Sandboxed scheduled runs cannot reach arbitrary network endpoints, and a usage-limited runtime cannot announce its own outage, because no run executes to send the alert.

**Options.**
- Keep the unofficial bridge as the rich channel - full capability now, terminal account risk accruing silently.
- The platform's official business/cloud API - compliant, but built for businesses messaging customers, heavy for one operator messaging himself.
- **A bot-platform channel on a first-party Bot API - official, free, HTML text plus documents, long-poll replies, one secret.**
- Email as the rich channel - durable and official, but the wrong cadence for a reply loop measured in minutes.

**Decision.** Five coupled moves.

1. **Official-API-first law.** A channel that requires an unofficial client is never load-bearing alone: it is mirrored by a canonical channel, degrades quietly (a send failure is a partial surface, never a run failure), is health-monitored by a dedicated probe, and is frozen the moment the platform pushes back, with the resume-or-retire call reserved to the human. New channels must ride an official API.
2. **Canonical + rich split, mirrored.** The OS-native text thread stays canonical: ordering, confirmations, and the reply loop of record. The bot channel carries what plain text cannot: HTML-formatted messages and file artifacts. Mirrors carry the same tag, content, and handles, reformatted per channel; reply processors poll every channel with per-channel watermarks and first-seen dedupe, so one channel's outage never stalls another.
3. **Outbox watch-folder senders (zero-LLM delivery daemons).** Agents do not send files; they drop them into a watched outbox directory. A launchd daemon on the host (60-second interval) performs the network send, verifies the API acknowledged, archives to a bounded sent/ folder, and skips files still being written (an mtime-settle check). One daemon per channel: the bot-API sender for message files and documents, and the text-thread attachment sender with HTML-to-PDF conversion, since the mobile client will not preview raw HTML. This moves network egress out of the sandboxed agent and leaves an inspectable queue instead of a lost send.
4. **Local Bot-API MCP.** A ~90-line stdlib-only stdio MCP server wraps the same Bot API as four tools (send message, send document, get updates, health), so interactive sessions get the channel as first-class tools while sandboxed scheduled runs use the outbox. Credentials live in one untracked local file read at call time; nothing is ever committed.
5. **Session-limit posture.** A usage-limit window kills runs at startup: ADR-11's scenario C with a known cause. The monitors classify it as one cluster (many tasks frozen in a single window, then a resume) instead of alerting N separate misses, and the last-resort notifier is zero-LLM: a bash + launchd sentinel that checks the fleet's run-marker file for staleness during waking hours and sends a plain OS-level message directly. No tokens, no model, works precisely when the model layer cannot.

**Consequences.** The rich channel is compliant and disposable-by-design (one secret, one daemon, one MCP file), and delivery no longer depends on the model being available or the sandbox having network reach. The mirror rule makes the notification layer device-independent. The honest residuals, monitored rather than hidden: the bot config is a single secret with no standby bot yet; the daemons live and die with the host being awake, so watcher liveness belongs in the fleet-health checks; and reply polling must be wired into the reply processor before the rich channel can carry commands, so until then it is send-only and the canonical thread carries all replies.

## Companion patterns adopted the same week

Recorded here rather than as separate ADRs, in the same compression. They extend [`design-patterns.md`](design-patterns.md).

**State dual-store.** Any task whose core promise depends on cross-run state (a diff baseline, a seen-log, a watermark) keeps it in two places: the primary vault file (full fidelity) and a device-note twin (replace-body, capped to what the next run needs). Reads go vault-first, note-fallback, and the output labels which store served. Failure semantics are explicit: a missing vault alone is a quiet partial when the note carries the state; both stores empty means first-baseline mode (reseed, no diff, no alarm); a loud failure is reserved for true state loss, both stores unreadable where a baseline previously existed. This converted a class of storage-outage false alarms into quiet degrades.

**Edit protocol for concurrent editors.** Two sessions once edited the same prompt and one overwrote the other. Convention since: every edit to an agent prompt or doctrine file prepends one edit-stamp comment line (timestamp, session, one-line summary); editors re-read immediately before writing and abort if an unseen stamp appeared (optimistic concurrency by hand); doctrine files are single-writer per sitting, with a proposals file for the second writer; and the nightly watchdog run spot-checks a rotating sample of prompts for drift. Cheap, grep-able, and a lost update shows in the next diff instead of disappearing.

**Temporal anchor + host-tz law.** Every scheduled run resolves time as its first action: UTC from a clock tool, operator-local from one tz override file, host-local from the fire context. Cron labels ride the host machine's current clock, so a travel day moves every fire time; operator-local reasoning rides the tz file; when the two diverge, the task translates explicitly and never assumes the label hour equals the local hour. No timestamp is reused across steps, and any output naming a day states the date.

**HITL board with handle verbs, confirmation-gated calendar writes.** The reply loop grew a board: on its full-mirror passes the reply processor renders every open item as a self-contained HTML board with per-board sequential handles, delivered on the rich channel with a one-line text prompt. The human replies handle + verb (done, push, drop, priority, block a time window, week for the 7-day view). The block verb creates a deliberately neutral calendar event (no sensitive titles on a shared calendar) plus a private prep reminder, both verified read-after-write; a bare same-day ok both confirms the current tiering and idempotently instantiates any still-proposed time blocks. All writes still flow through the one reply-processor lane, so the board adds reach without adding a writer.

**Intake queue with a single consumer.** Upstream watchers append one JSON line per net-new signal to an append-only queue file; writers never edit or delete. Exactly one consumer reads it, dedupes against its own seen-log, promotes qualifying items, and, as the queue's sole consumer, alone compacts the file to a rolling window at the end of a successful run (never on a degraded run, so nothing is lost mid-outage). Many writers, one compactor: append-only semantics for everyone else.

**Model pin + policy, two layers.** Model choice splits into an intent layer and an execution layer. The intent layer is a pin token in each task's description: the model the human sets in the app's per-task dropdown, human-applied and human-owned, the auditable record of what should be. The execution layer is the in-prompt delegation policy line (synthesis on the main model, mechanical fan-out on a mid tier, evaluators on the cheapest), owned by a dedicated controller that may adjust it only inside a safety envelope: surgical single-line edits with full-prompt backups, post-write verification with rollback, a per-run mutation cap, a protected core it may never touch (the independent watchdog and itself), and same-day loud disclosure of every change. Two layers, two owners, one field each: the fence law applied to model selection itself.

## Addendum - channel hardening (2026-07-08 to 2026-07-09)

Two more days on the same channel-risk thread produced four coupled hardening moves. Appended here rather than filed as a new ADR, because they tighten this decision rather than replace it.

**(a) Parity v3 - one canonical, one rich, mirrored 1:1.** The channel model simplified from a three-way mirror to a deliberately narrower two-leg parity: one canonical plain channel (SMS-class - ordering and confirmations stay here) mirrored one-to-one into one rich channel (bot chat: HTML formatting plus file delivery). Both legs carry the same tags and the same handles; only the formatting is per-channel. Fewer legs, the same coverage, less surface to keep in sync.

**(b) Inbound auth gate.** A public-facing bot is a public-facing command surface, so every inbound command is checked against a pinned chat id read from local, untracked config before it executes. A message from any other chat is logged and never executed. This is defense against a specific failure mode: a public bot username becoming a command-injection path. An unrecognized sender is not a user the system trusts, no matter what it asks for.

**(c) Full-pause plus staged re-entry ladder.** When a third-party channel's own risk model starts to look like it could trigger account enforcement, the response is to pause every leg on that channel silently and all at once, so there is no partial-surface noise while the risk is live. Re-entry is never a single flip back on. It climbs a ladder gated on a clear verdict: receive-only for 14 days, then critical-alerts-only for 7 days, then full text mirrors, then file delivery last - and the channel's own health monitor must show green again before any rung of the ladder is allowed to send.

**(d) Outbox-file fallback.** A sandboxed run cannot always reach a messaging API directly. When it can't, the task drops a fully formatted file into a watched outbox directory instead of failing the send; a small host-side daemon (launchd, a 60-second poll) picks it up and delivers it. The file drop needs no network connector and no live credential at write time, so a mirror that would otherwise vanish quietly instead waits in a folder until the daemon, or a human, can ship it.
