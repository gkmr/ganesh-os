The governance layer · the hard part

The guardrail is enforced, not asserted.

Every mutable field has exactly one owning agent. An agent literally cannot write a field it does not own, because a behavioral eval over the change log blocks it in CI. Here's the actual check:

evals/lane_fence.py
# every mutable field has exactly one owning agent
OWNER = {
    "priority":    {"pipeline-triage", "todo-triage"},
    "due_date":    {"morning-sweep", "evening-sweep"},
    "lifecycle":   {"reply-processor"},   # create / complete
    "delete":      set(),                 # nobody: human-gated
}

def check_lane_fence(changelog):
    for e in changelog:
        assert e.agent in OWNER[e.field], \
            f"CROSS-LANE: {e.agent} wrote {e.field}"
    return "PASS"   # a regression blocks the merge, rolls back

📐 System design · an engineering leader's pass

Business goal first, then requirements, scale, interface, architecture, deep dive, and what v2 needs. AI as the protagonist, not an ingredient.

Step 1 · Requirements
KPI → minutes-to-first-right-action each morningCollapse a multi-tool scan into one ranked text, and never let a loud domain starve a quiet one. Everything downstream serves this.
FunctionalIngest every channel; tier and rank; reconcile calendars; deliver one cross-domain brief; accept a reply from any surface and apply it safely; grade the day.
Non-functionalRuns unattended on an intermittently-awake host; auditable and recoverable; safe to act on everything reversible; one operator, global travel, multi-channel.
CAP choicePer field, consistency over availability: one owner, no lost writes. Across the system, availability via degradable surfaces and idempotent catch-up. Partition-tolerant by design (files).
Scoping — explicitly outMulti-user/tenant, auth, real-time collaboration, and a query engine are deliberately out of scope for v1. In scope: one operator, batch (cron) cadence, correctness, and auditability. Naming what's out is half the senior answer.

High level first: connectors feed capture, triage owns priority, the sweeps own dates, the brief ranks and delivers, the human replies from any channel. The change log is the spine; evals are the gate; memory is re-hydrated every run.

def brief(canvases, calendar, messages):

Scaling the next 10×If grep stops scaling, a local SQLite index sits behind the same file interface. The pattern generalizes to a team without changing the fences.

Step 7 · Reliability — retry, fallback & model policy

The interesting engineering is the unhappy path and the model boundary. Four policies, stated plainly:

Retry = re-fire, not a queueNo blind retries against shared state — that is how you get double-writes. A concurrency guard + run-ledger make every run safe to repeat, so the schedule itself is the retry: the next slot re-derives the world from disk and closes what the last one left open. A within-run connector failure backs off, then degrades rather than aborting.
Fallback: surface → reduced → deferA surface check runs each agent on whatever connectors are present, labels the output "partial surface," and queues write-intents for the next full run. Completeness flexes under degradation; the single-writer fence and the audit log never do.
Model policy: judgment vs. determinismAnything that must be exact — ids, calendar math, the per-day budget, the lane-fence — is deterministic code. Model judgment is confined to the fuzzy: surface the three real messages, rank a day, write coaching. The model proposes; code disposes of anything irreversible.
Tiered model routing + fallbackThe cheapest model that can do the job: a small fast model for capture and formatting, a stronger one for triage, reconciliation, and coaching — declared per agent. If the preferred model is down, the agent drops a tier and records which model ran; reasoning output marks itself reduced-confidence rather than blocking. The harness owns the model boundary, so the model is a swappable part.
Step 8 · Privacy & security

It runs on the most sensitive data a person has — every message, calendar, and health signal. So privacy and security are architectural, not bolted on:

Local-firstThe knowledge base, verbatim archives, and change log are files on the operator's own device. State leaves the machine only through the operator's own connectors, into their own accounts — never a third-party server.
Least privilege, by constructionEach connector is scoped to what its agents need; each agent reads only what its job requires and writes exactly one field. The single-writer fence is least-privilege for writes — touching state you don't own is a CI failure, not a code-review note.
The human gate is an exfiltration controlEvery outbound or irreversible action is gated on an explicit human decision. An agent can't autonomously send data out — the same gate that stops mistakes stops exfiltration.
No secrets, no PII in the openCredentials are never committed; the public repo is architecture and sanitized patterns only, enforced by a pre-publish scan and a .gitignore that fences the live system, with secret scanning on as a backstop.
Auditability is a security propertyThe append-only, source-tagged change log makes every action attributable. What touched this, when, and why is one grep — the difference between a system you can secure and one you can only hope about.
Decisions & verdicts (ADRs)
ADR-01 · How do 27 writers not clobber each other?
✓
Single-writer field fences, enforced by a one-rule eval.
✕
A transaction log / MVCC — textbook, but the store has no transactions.
Verdict: same safety with zero infrastructure, auditable by one check. Cost: a convention enforced by CI, not the platform.
ADR-02 · Where does the state live?
✓
Plain Markdown files — a model and a human can both read and hand-edit from a phone.
✕
SQLite / Postgres — real queries, but every run is stateless and edited from a phone.
Verdict: wins on legibility and zero migration. SQLite index is the planned next step if grep stops scaling.
ADR-03 · One big agent, or many?
✓
27 small agents — a bug degrades one function; the schedule is the orchestration.
✕
A mega-agent — no coordination, but full blast radius and no observability.
Verdict: wins on blast radius. Cost: the coordination layer — the part worth showing.
ADR-04 · Wait for confirmation, or move?
✓
Auto-park the reversible; gate only the irreversible.
✕
Confirm every change — safer, but produced a permanent backlog.
Verdict: forward progress by default. Cost: an item can move without a say-so; mitigated by tiers, sacred exclusions, a report.

The memory is the moat

Longitudinal context, stored without rot and recalled without lying.

Proprietary, long-running data is only an advantage if memory compounds instead of drifting. The design borrows three ideas and makes them run: MemPalace validity windows, the Karpathy LLM-wiki substrate, and a second-brain layer I call gbrain.

MemPalace
Verbatim store + temporal validity windows

A fact is never silently overwritten. It carries an "as of" and a "superseded by" stamp, so the coach remembers you were sick three weeks ago and adjusts, then forgets it when it stops being true.

Karpathy LLM-wiki
raw · wiki · index · log

A read-first index over a compiled knowledge layer, so an agent loads a small routing map before opening any full file. Explicit ingest, query, and lint passes keep it honest.

Proof the pain is real

Four real failures, each traced symptom → root cause → durable fix.

Names and data are generalized; the engineering is exactly as it happened. Each is a governance decision, and each ends in an eval so the class of bug can't return.

CASE 01

The overdue collapse: 99 → 0, held there

A polite system that waited forever vs. a loop that has to advance.

Symptom
99 in "today and overdue," a 63-item pile that grew no matter how often it was triaged.
Root cause
Agents waited for a confirmation before moving any date, so nothing moved. A snapshot silently skipped two lists.
Fix
Auto-park re-dates the undecided by tier; triage now enumerates every list dynamically.
Gate the irreversible; let the reversible flow.
CASE 02

The single-day pileup

Correct tiers, wrong shape — a pile by another name.

Symptom
After a bulk re-tier, one day held ~48 tasks while neighbors sat empty.
Root cause
No per-day capacity; recurring rituals counted as tasks; ranked purely by urgency.
Fix
A per-day budget, a spread mandate, and one top-three slot per domain.
Prioritization without distribution is just a different pile.
CASE 03

From stream to action surface

Awareness without throughput is a louder badge.

Symptom
Five channels produced "you have an unread thing." Each still meant switching apps.
Root cause
Digests summarized well but dead-ended. No path to "done" without leaving the phone.
Fix
A manifest handle per item; reply "done 3" and an hourly processor applies it with read-after-write verify.
A notification is only useful if the reply path is as cheap as the alert.
CASE 04

The stale id: a silent write that did nothing

The difference between a demo and a system you trust with a life.

Symptom
Writes failed with record-not-found. A handful of "completes" silently did nothing.
Root cause
An id reproduced from memory across steps pointed at nothing.
Fix
Read every id fresh before a write; verify by id; a not-found id is a hard error, fenced by CI. CLOSED.
Silent failure is the only unforgivable one. Make it loud, then fence it.
